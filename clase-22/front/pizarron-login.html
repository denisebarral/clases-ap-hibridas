❓ Tu pregunta: ¿Por qué call() manda el token si el login no necesita token?
Respuesta corta: porque call() no sabe para qué ruta se va a usar. Es una función genérica que siempre arma el mismo header. Para el login, el token es null (el usuario todavía no se logueó), así que el header queda Authorization: Bearer null.

¿Y el backend se rompe por eso? No. La ruta /api/login no tiene el middleware validateToken, así que el backend ignora completamente ese header. Es como escribir un campo en un formulario que el servidor ni lee.

✅ Conclusión: call() manda el token siempre. Para rutas públicas (login, registro) no importa. Para rutas protegidas (libros) es necesario. Una sola función, dos usos.
📋 Paso a paso: el login completo
1
El usuario aprieta "Ingresar" Login.jsx
Se dispara handleSubmit(). Se leen email y contraseña del formulario.
const email = e.target.email.value // "denise@mail.com"
const pass = e.target.pass.value // "Clave123!"
2
Login.jsx llama al service usuarios.service.jsx
Login.jsx NO hace fetch por su cuenta. Le pide al service que lo haga.
loginService({ email: "denise@mail.com", password: "Clave123!" })
🎯 ¿Por qué un service y no fetch directo? Para que Login.jsx no sepa CÓMO se llama a la API. Solo sabe QUÉ quiere: hacer login. El CÓMO es problema del service.
3
usuarios.service llama a api.service api.service.jsx
usuarios.service tampoco hace fetch. Llama a call() del api.service.
// En usuarios.service.jsx:
const login = (credenciales) => call("/login", "POST", credenciales)

// call() recibe: uri="/login" method="POST" body={ email, password }
4
api.service hace el fetch api.service.jsx
Acá sí sale la llamada HTTP real. call() arma el fetch con la URL completa y los headers.
fetch("http://localhost:2026/api" + "/login", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": "Bearer null" // ← token es null (aún no logueado)
  },
  body: JSON.stringify({ email, password })
})
⚠️ El token es null porque viene del Contexto y el usuario aún no se logueó. El header queda "Bearer null" pero el backend NO lo lee para la ruta /api/login (no tiene el middleware validateToken). No pasa nada.
5
El backend procesa el login Back — Express
La request llega al back. Las capas que se ejecutan en orden:
POST /api/login
↓
validateLogin (middleware Yup) → verifica que email y password tengan formato correcto
↓
login() controller → llama al service
↓
login() service → findOne({ email }) en MongoDB
↓
bcrypt.compare("Clave123!", "$2b$11$hash...") → true ✅
↓
createToken(usuario) → jwt.sign con SECRET_PASSWORD del .env
↓
← devuelve { email, token: "eyJhbGci..." }
6
api.service recibe la respuesta y la devuelve
El then() de api.service verifica que fue exitosa (res.ok) y parsea el JSON.
.then(res => {
  if (res.ok) return res.json() // → { email: "denise@mail.com", token: "eyJ..." }
  // si fuera 401, redirigiría al login (no aplica acá)
})
7
Login.jsx recibe los datos y llama al Contexto useLogin()
Con los datos del backend, Login.jsx actualiza el Contexto global de sesión.
.then(data => {
  // data = { email: "denise@mail.com", token: "eyJhbGci..." }
  login(data.token, email) // ← función del CONTEXTO
  navigate("/") // ← redirige al Home
})
8
El Contexto guarda la sesión Session.context.jsx
onLogin() hace dos cosas: guarda en localStorage (para que sobreviva al reload) y actualiza el estado de React (para que los componentes se re-rendericen).
const onLogin = (jwt, usuario) => {
  localStorage.setItem("token", jwt) // persiste al reload
  localStorage.setItem("session", ...) // persiste al reload
  setToken(jwt) // → api.service ahora tiene el token real
  setEmail(usuario) // → NavBar se re-renderiza mostrando "Salir"
}
🎯 Después de este paso, el token real ya está en el Contexto. La próxima vez que api.service llame a call(), el header será "Authorization: Bearer eyJhbGci..." (el token real, no null).
9
navigate("/") → Home carga los libros
El usuario llega al Home. Home llama a getLibros(), que usa call() del api.service. Esta vez el token SÍ es real (ya fue guardado en el Contexto en el paso 8).
// Home.jsx
const { getLibros } = useLibrosService()
getLibros() // → call("/libros") con token real → back valida → devuelve libros
🗂️ ¿Quién sabe qué? — Mapa de responsabilidades
📄 Login.jsx / Home.jsx / etc.
Solo sabe QUÉ quiere hacer.
No sabe CÓMO se llama a la API.
No sabe DÓNDE vive el token.
🔧 usuarios.service / libros.service
Sabe QUÉ endpoints existen.
"/login", "/libros", "/libros/:id"
No sabe cómo se arma el fetch.
🌐 api.service
Sabe CÓMO se llama a la API.
URL base, headers, token, 401.
No sabe para qué ruta se usa.
🧠 Session.context
Guarda el token y el email.
Reactivo: cuando cambia, todos
los componentes se actualizan.
⚖️ Antes (clase-20) vs Ahora (clase-22)
❌ Antes — todo en el componente
// En Home.jsx
fetch("http://localhost:2026/api/libros", {
  headers: {
    authorization: `Bearer ${localStorage.getItem("token")}`
  }
})
.then(res => {
  if (res.ok) return res.json()
  if (res.status == 401) navigate("/login")
  throw new Error("...")
})
.then(data => setLibros(data))
✅ Ahora — responsabilidades separadas
// En Home.jsx
const { getLibros } = useLibrosService()

getLibros()
  .then(data => setLibros(data))
